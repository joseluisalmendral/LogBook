# LogBook v1.3 Roadmap — Distribution + Self-Update + Auxiliary Docs

**Status**: deferred from v1.2 by user decision. To be tackled "with calm" in a future release window.

This document is the **canonical record** of what was discussed, what was decided, what's left, and exactly what needs to happen — including any credentials / accounts the maintainer must provision before work can start.

When you (or another agent) come back to this, **read this top-to-bottom** before opening any code. The decisions below were made on 2026-05-18 in a long conversation; they're locked unless someone explicitly revisits them.

---

## TL;DR

v1.3 is about **shipping LogBook as a polished CLI** that anyone can install in one command:

```bash
# macOS / Linux
brew tap joseluisalmendral/tap
brew install logbook

# Windows
scoop bucket add logbook https://github.com/joseluisalmendral/scoop-bucket
scoop install logbook
```

Plus the supporting machinery: `logbook self-update`, startup-time update check, `SECURITY.md`, `docs/privacy.md`, and a CLI bundle refactor to reclaim headroom (we're at ~170 bytes from the 400 KB cap after v1.2).

Estimated scope: ~600 LoC + 2 new GitHub repos + 1 GitHub Actions workflow + 1 PAT secret.

---

## Section 1 — Why this is its own release

When the user asked for v1.2 they reduced scope to **only the in-tree improvements** (streaming LLM, MCP clock injection, doctor bundle warning). The reasoning:

> Tengo un poco de prisa al final, añade solo las mejoras y lo demás ya lo vemos con más calma más tarde.

Distribution + self-update + auxiliary docs are a **fundamentally different kind of work**:

- They require operational decisions (which org owns the tap, what PAT scopes, codesigning yes/no)
- They require creating GitHub artifacts (repos, secrets) outside the LogBook repo
- They benefit from being tested manually on a real macOS / Linux / Windows machine before tagging a release

v1.3 is the window to do that properly.

## Section 2 — Scope

### 2.1 — Distribution: brew + scoop + standalone binaries (~400 LoC)

#### What ships

End-state user experience:

```bash
# macOS arm64 / x86_64 and Linux x86_64 / arm64
brew tap joseluisalmendral/tap
brew install logbook

# Windows x86_64
scoop bucket add logbook https://github.com/joseluisalmendral/scoop-bucket
scoop install logbook
```

After install, the user has a `logbook` binary in `$PATH` with no Node runtime required.

#### Architecture decisions (locked)

- **Binary packager**: `@yao-pkg/pkg` (mature fork of Vercel's deprecated `pkg`). Reasons:
  - Handles native deps (`better-sqlite3`, `puppeteer-core`) out of the box
  - Cross-compiles to 5 targets from a single host
  - Battle-tested by Volta, older Vercel CLI versions
  - Node 22 SEA was evaluated but rejected: `better-sqlite3` requires manual `.node` extraction workaround that breaks on every Node upgrade
- **Platforms** in build matrix:
  - `darwin-arm64` (Apple Silicon macs)
  - `darwin-x64` (Intel macs)
  - `linux-x64`
  - `linux-arm64`
  - `windows-x64`
- **Distribution**: GitHub Release assets, one tarball per platform, SHA256 in `.checksums.txt`
- **Codesigning**: SKIP for v1.3. Users see "developer not verified" warnings on macOS / SmartScreen on Windows. Acceptable for first public release. Can add later if budget / Apple Developer account becomes available.

#### Repositories needed

Three repos under `joseluisalmendral`:

| Repo | Purpose | Contents |
|------|---------|----------|
| `LogBook` (existing) | Main project | Source + CI release workflow |
| `homebrew-tap` (NEW) | Homebrew tap | `Formula/logbook.rb` |
| `scoop-bucket` (NEW) | Scoop bucket | `bucket/logbook.json` |

The Formula and Manifest files are **auto-generated and pushed by the CI workflow** on each release. The maintainer creates the repos empty and grants write access to the PAT.

#### Credentials needed (the human-to-do list)

1. **Create `homebrew-tap` repo** under `joseluisalmendral`:
   ```bash
   gh repo create joseluisalmendral/homebrew-tap --public --description "LogBook Homebrew formula"
   ```
2. **Create `scoop-bucket` repo** under `joseluisalmendral`:
   ```bash
   gh repo create joseluisalmendral/scoop-bucket --public --description "LogBook Scoop manifest"
   ```
3. **Generate a fine-grained Personal Access Token**:
   - URL: <https://github.com/settings/personal-access-tokens/new>
   - Resource owner: `joseluisalmendral`
   - Repository access: `LogBook` + `homebrew-tap` + `scoop-bucket`
   - Permissions:
     - `Contents`: Read and Write
     - `Workflows`: Read and Write
     - `Metadata`: Read (implicit)
   - Expiration: 1 year (set a calendar reminder to rotate)
4. **Store PAT as a repo secret in `LogBook`**:
   - Settings → Secrets and variables → Actions → New repository secret
   - Name: `RELEASE_PAT`
   - Value: the token from step 3

#### Possible name conflict

There is an existing Python `logbook` (logging library by Armin Ronacher). It MAY conflict with `brew install logbook` short form. Verification path:

```bash
brew search logbook
```

If `logbook` is already in `homebrew-core`, options are:
- Use the explicit tap form: `brew install joseluisalmendral/tap/logbook` (always works, slightly verbose)
- Rename the formula to `logbook-cli` (clean shortcut but diverges from package name)
- Keep `logbook` but document the explicit form in install instructions

Decide at the moment of writing the Formula.

#### Implementation outline

1. **`scripts/build-binaries.ts`** (~150 LoC) — driver that invokes `@yao-pkg/pkg` for each target, computes SHA256 of each output, writes `dist-binaries/checksums.txt`.
2. **`.github/workflows/release.yml`** (~200 LoC) — triggered on `git push --tags`. Job matrix per platform. Steps:
   - Checkout
   - Setup Node 22
   - `pnpm install --frozen-lockfile`
   - `pnpm test` (full suite)
   - `pnpm build`
   - `pnpm exec tsx scripts/build-binaries.ts <platform>`
   - Upload artifact
   - After all platforms succeed: create GitHub Release with artifacts, generate Formula `.rb` from template, push to `homebrew-tap`; same for Scoop `.json` → `scoop-bucket`
3. **`scripts/templates/formula.rb.ejs`** + **`scripts/templates/scoop.json.ejs`** — checksum-aware templates
4. **`docs/08-distribution.md`** (NEW) — operator guide: how to bump version, how to trigger a release, how to debug failed releases

### 2.2 — `logbook self-update` command (~80 LoC)

#### Purpose

Even with `brew` and `scoop`, users don't run `brew upgrade` regularly. A built-in `logbook self-update` gives parity with tools like `gh`, `deno`, `cargo` that have first-class update flows.

#### Behavior

```
$ logbook self-update
Current version: 1.2.0
Checking https://api.github.com/repos/joseluisalmendral/LogBook/releases/latest...
Latest version: 1.3.2
Downloading logbook-darwin-arm64-1.3.2.tar.gz... (12.4 MB)
Verifying SHA256... ok
Replacing /usr/local/bin/logbook... done
✓ Updated to 1.3.2
```

#### Implementation

- New CLI subcommand `src/cli/commands/self-update.ts`
- Fetch from GitHub Releases API: `GET /repos/joseluisalmendral/LogBook/releases/latest`
- Detect current platform via `process.platform` + `process.arch`
- Download tarball + verify SHA256 against `.checksums.txt` published in the release
- Atomic swap: download to temp dir → verify → rename current binary to `.bak` → move new in place → on failure rollback from `.bak`
- Refuse to run if installed via `brew` / `scoop` (detect by parent directory in `process.execPath`) and suggest `brew upgrade logbook` instead — package managers should manage their own packages
- Flag `--prerelease` to opt into rc / beta versions

#### Caveats

- macOS unsigned binaries require `xattr -d com.apple.quarantine` after download — the command should do this automatically with a clear message
- Windows: file replacement requires releasing the running executable; on Windows, schedule a `.bat` to replace after process exit
- If signing is added later (v1.4+?), the verification step also checks signature

### 2.3 — Update check on startup (~50 LoC)

#### Purpose

Most users don't run `self-update` proactively. A non-invasive check on startup nudges them.

#### Behavior

When the TUI shell or any CLI command runs:

1. Skip if `LOGBOOK_NO_UPDATE_CHECK=1` is set
2. Skip if the last check was less than 24 hours ago (timestamp in `.logbook/state.json`)
3. Fire an async HTTP request to GitHub Releases API with a 2-second timeout
4. If a newer version exists, append a dim footer line to the next output:
   ```
   v1.3.2 available — run `logbook self-update` to upgrade
   ```
5. Cache the latest version + check timestamp in `state.json` for 24h

#### Implementation

- New module `src/cli/update-check.ts`
- Background fire-and-forget: never blocks the user's command
- Failure is silent (network down, GitHub rate-limited, etc.) — don't spam errors
- Tests: mock fetch + filesystem + verify cache TTL

#### Privacy note

The request is `GET https://api.github.com/...` with no body. GitHub sees the IP address and User-Agent (we send `LogBook/<version>`). No telemetry. The `docs/privacy.md` documents this.

### 2.4 — `SECURITY.md` (~30 lines)

#### Purpose

Standard for distributed CLI tools. Tells security researchers how to report vulnerabilities privately (not via public GitHub Issues).

#### Contents

```markdown
# Security Policy

## Supported versions

| Version | Supported |
|---------|-----------|
| 1.2.x   | ✓         |
| 1.1.x   | ⚠ critical fixes only |
| < 1.1   | ✗ |

## Reporting a vulnerability

Email <security@redacted> with:
- A description of the issue
- Steps to reproduce
- Affected version(s)
- Suggested fix if you have one

We'll respond within 72 hours. Please do NOT open a public GitHub Issue
for security reports.

## Scope

In scope:
- Path traversal / sandbox escape from MCP tools
- Credential leakage in JSONL / generated docs
- XSS in HTML export
- Code execution via theme CSS / mermaid SVG

Out of scope:
- DoS via large JSONL (use rate limiter)
- Vulnerabilities in optional Chrome runtime
- LLM provider security (out of our control)
```

#### Credentials needed

- An email address for the security contact. Options:
  - Personal email (simplest)
  - Dedicated `security@<your-domain>` (more professional, requires owning a domain + mail setup)

### 2.5 — `docs/privacy.md` (~80 lines)

#### Purpose

Critical for pedagogical use: instructors and students need to know **where does my code go** when they use LogBook.

#### Outline

1. **What LogBook collects** — nothing on remote servers. All data lives in `logbook/` inside the user's project.
2. **What LogBook sends to LLM providers** — when `summarize` / `decision` etc. are invoked, the configured provider receives:
   - The events in the relevant time window (redacted via `sanitizeForSafeExport` if `--safe`)
   - The system + user prompt
   - No other context
3. **What the LLM provider sees** — depends on provider:
   - Anthropic / OpenAI / Google: per their respective privacy policies
   - Ollama (local): nothing leaves the machine
   - Codex CLI: per Codex's policy
4. **What goes to GitHub** — only the update-check API call (no body, no user data). Opt-out via `LOGBOOK_NO_UPDATE_CHECK=1`.
5. **Data retention** — local only. Uninstall removes nothing inside `logbook/`; manual deletion required.
6. **Children's data** — LogBook is not designed for use with users under 13. Instructors should review provider privacy policies before using LogBook with student work.

### 2.6 — CLI bundle refactor (~100 LoC)

#### The problem

After v1.2, `dist/cli/index.cjs` is at **399.83 KB / 400 KB**. The SG-C soft warning is firing on the bundle itself (>380 KB). Any new feature touching the cold-start path will likely exceed the hard cap.

#### Root cause

The CLI bundle includes the TUI shell code (React + Ink + screens + components). The TUI is lazy-loaded via `await import()` at runtime when the user invokes `logbook` with no args, but tsup with `bundle: true` still includes the code in the main bundle — it doesn't actually split the chunk for CJS output.

#### Approach options

| Option | Effort | Gain | Risk |
|---|---|---|---|
| **A. Move TUI to its own tsup entry** (`dist/tui/shell.cjs`) and require it via `require()` from CLI | Medium | -40 to -60 KB | Low — same dynamic-import semantics, just real file split |
| B. Migrate CLI to ESM | High | -20 KB | High — many transitive deps still CJS-only |
| C. Split each TUI screen into its own bundle | High | -30 KB | Medium — startup penalty per screen mount |

**Recommendation**: Option A. Adds one tsup entry + one `require()` call. Same behavior, real bundle split.

#### Implementation outline

1. `tsup.config.ts`: add `dist/tui/shell.cjs` as the 6th entry
2. `src/cli/index.ts`: change `await import("../tui/shell.js")` to `require("../tui/shell.cjs")` (or the equivalent that works post-bundle)
3. Externalize React + Ink + ink-* in the CLI entry so they only live in the TUI bundle
4. Verify byte-identity gates remain green (TUI never touches install/uninstall surface anyway)
5. Update `cli-bundle-size.test.ts` to assert new lower upper bound (e.g., 360 KB)

## Section 4 — Non-goals for v1.3 (explicit)

- **Codesigning** (macOS Apple Developer Program $99/yr, Windows code signing cert $200+/yr). Acceptable to ship unsigned in v1.3. Add when there's demand or budget.
- **Telemetry** — opt-in anonymous usage stats. Useful but adds a privacy surface. Reconsider in v1.4.
- **SLSA provenance attestations** — overkill until LogBook has a security-conscious user base.
- **`apt` / `dnf` packages** — Debian/RPM packaging is its own multi-week project. Skip unless Linux distros become a primary user segment.
- **Migration guide from `pnpm install -g`** — existing users (small base) can be told "uninstall via pnpm, install via brew" in CHANGELOG.

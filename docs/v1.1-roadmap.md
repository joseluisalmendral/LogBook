# LogBook v1.1 Roadmap — Post-MVP Backlog Analysis

The MVP (iter1 through iter5) and the unified TUI shell (iter6) are feature-stable on `main`. This document inventories every deferred item from the six iteration archive reports, scores each one by value for an instructor / teaching use case, and recommends which deserve a v1.1 slot, which can wait for v1.2+, and which should be left alone. The user starts using LogBook in production for instructor work in a few days, so the bar for shipping is "does this change what the instructor or the audience actually sees or does", not "does this round out the matrix".

## Quick read

Five-line summary of the recommended v1.1 cut:

1. **Item 1 — `rehype-slug` for anchor navigation** in `instructor-pack.html`. The single most visible quality gap; TOC links currently look clickable but go nowhere.
2. **Item 8 — `retry: 2` for `mcp-rate-limit.test.ts`** in `vitest.config.ts`. Two-line fix that removes a flake that has shadowed every iter since iter3.
3. **Item 9 — extend `--safe` to `logbook build`** so the artifacts inside `logbook/` are redactable, not just the HTML exports.
4. **Item 7 — animated spinner in DoingScreen**. Tiny but visible polish in the TUI; instructor demos run through this surface.
5. **Item 20 — `init --upgrade` UX**. Right now re-running `init` on an installed project silently shows skip lines; users in production need a clean upgrade path.

Everything else either is invisible to the audience (Items 4, 5, 6, 11, 13, 14, 18), too costly for the window (Items 2, 3, 15, 17), or needs more usage-driven data before committing (Items 10, 12, 16, 19).

## Section 1 — Methodology

Each backlog item is scored along two primary axes and three secondary factors.

### Primary axes

- **Value (1–5)** — impact on the instructor / teaching use case.
  - **5** = changes what students or stakeholders see in the final output (HTML, ADRs, teaching scripts) OR removes a visible production-time blocker.
  - **4** = significant usability improvement that the instructor will notice on every session (TUI ergonomics, install flow, build flow).
  - **3** = a real but bounded quality improvement (test stability, modest perf, doc polish that lowers support load).
  - **2** = infrastructural benefit only the maintainer sees.
  - **1** = invisible to end user; infra-only or future-coupling.
- **Cost (1–5)** — implementation effort.
  - **1** = ≤ 100 lines + ≤ 1 day, no new deps.
  - **2** = ≤ 250 lines + 1–2 days, possibly 1 small dep.
  - **3** = ≤ 500 lines + 2–4 days, 1 dep that needs vetting.
  - **4** = ≤ 1000 lines + 4–7 days, ≥ 1 dep with non-trivial surface (puppeteer, mermaid-cli, sql.js, node-pty).
  - **5** = ≥ 1000 lines + week+, multiple deps or architectural rework.

### Secondary factors

- **Risk** — Low / Medium / High. Probability of regressing existing features (byte-identity gates, token budget, hook p95, no-real-LLM-in-CI).
- **Scope creep** — Yes / No. Does shipping it pull in new conceptual surface (a new format, a new daemon, a new permissioned operation)?
- **Pre-condition** — None or "blocked by Item N".

### Priority score

`priority = Value × (6 − Cost)`

Rationale: the formula rewards items that move the user-facing needle and penalizes items proportional to their effort. A Value-5 / Cost-1 item scores 25. A Value-2 / Cost-5 item scores 2. Anything ≥ 15 is a strong v1.1 candidate; 8–14 is a maybe; below 8 should defer unless a blocking dependency forces it.

The formula intentionally over-weights cheap visible wins. The user is heading into production this week and needs a tight, focused v1.1, not a clean-up sweep.

## Section 3 — Per-item analysis

### Item 1 — `rehype-slug` for anchor navigation in `instructor-pack.html`

**Source**: iter5 verify report obs #122 W2; carried in iter6 W2 unchanged.
**Spec reference**: §40 ("Output profesional HTML"), §27 (generación de documentos).
**Description**: The TOC at the top of `instructor-pack.html` renders as a list of hyperlinks pointing at `#sections`, but the `<a id="...">` anchor tags that `instructor-pack.ts` emits get stripped by `rehype-stringify` when `allowDangerousHtml: false`. Adding the `rehype-slug` plugin to the unified pipeline regenerates IDs on headings deterministically and makes the navigation work.
**Value rationale (5)**: This is the artifact the instructor hands to stakeholders or shares with students. A broken TOC reads as "this tool is half-finished". Fixing it changes what every viewer of the HTML pack experiences on first scroll. No other backlog item has the same direct visibility-to-audience ratio.
**Cost rationale (1)**: Add `rehype-slug` as a dep (one well-maintained package), plug it into the pipeline in `src/export/html.ts` and `src/export/instructor-pack.ts` (single line each), drop the manual `<a id>` injection in `instructor-pack.ts` (~10 lines removed), add one integration test asserting `#section-id` exists in output. Total: ~30–50 lines plus 1 dep.
**Risk**: Low. `rehype-slug` is a pure transform on the hast tree; no DOM, no runtime. The byte-identity gates do not test the export bundle (they test the install artifacts), so no GATE regression risk. The export bundle's "0 external refs" assertion is unaffected because rehype-slug doesn't inject external resources.
**Pros**:
- Closes the most-mentioned warning across three iterations.
- Restores a feature that the instructor will use the moment they share the pack.
- Zero conceptual surface added; the TOC contract is already documented.
**Cons**:
- One new transitive dep (rehype-slug pulls in github-slugger).
- Slug collisions if two headings share the same text need a tie-breaker policy — `rehype-slug` handles this by appending `-1`, `-2`, but the TOC link generator in `instructor-pack.ts` needs to match the same algorithm. This is the only non-obvious work.
**Recommendation**: **SHIP IN v1.1**.
**If shipped**: One slice. `pnpm add rehype-slug`, wire into both export pipelines, replace manual anchor injection with `github-slugger` import to align TOC link generation with rehype-slug's output, one integration test.

### Item 2 — `instructor-pack.pdf` export via puppeteer or pandoc

**Source**: iter5 verify report obs #122 S1; iter5 roadmap; carried in iter6.
**Spec reference**: §35 iter5 row ("`instructor-pack.html` y `.pdf`"), §10 (`logbook export html|pdf|instructor-pack`).
**Description**: Spec §35 lists `.pdf` as an iter5 deliverable alongside `.html`, but it was deferred because the implementation requires bringing in either Puppeteer (~300 MB Chromium download on first run) or Pandoc (system dep with LaTeX for proper styling). The CLI signature in §10 already reserves the slot (`export html|pdf|instructor-pack`).
**Value rationale (3)**: Stakeholders often want a PDF for offline reading or attaching to email. For an instructor, this is the second-most-shareable artifact after the HTML. But the HTML pack is fully self-contained and printable from any browser to PDF, which covers ~80% of cases. The "browser print to PDF" workaround is mentioned in the troubleshooting docs.
**Cost rationale (4)**: Puppeteer adds a 300 MB Chromium download, which is incompatible with the "students can READ the source" philosophy and roughly doubles install time for new users. Pandoc is system-level — out of scope for a pnpm-installed CLI. A middle path is `puppeteer-core` + bring-your-own Chrome, but that pushes user friction onto every install. Realistic effort: 4 days, ~500 lines, plus one heavy dep and config docs.
**Risk**: Medium. The first Puppeteer install can fail in restricted environments (CI sandboxes, corporate VPN). The CJS bundle size impact is significant if we don't lazy-load. Lazy-loading is doable but adds an entrypoint.
**Pros**:
- Closes a spec-listed gap.
- Real user value for distribution.
**Cons**:
- Heavy dep relative to all current bundles combined.
- The browser print-to-PDF workaround is good enough for the immediate use case.
- High maintenance burden (Chromium pinning, headless flag drift across versions).
**Recommendation**: **DEFER TO v1.2** (or skip until a user complains). The print-to-PDF workaround removes urgency.
**If shipped**: A separate slice on its own, after v1.1 ships. Add a fifth tsup entry `dist/export/pdf.cjs`, lazy-load puppeteer-core, document the bring-your-own-Chrome path. Estimate 1 week.

### Item 3 — Mermaid diagram rendering via `mermaid-cli`

**Source**: iter5 verify report obs #122 S2; carried in iter6.
**Spec reference**: §34 (Mermaid listed as **opcional**), §40.
**Description**: Fenced ```mermaid blocks in ADRs or teaching scripts currently render as `<code>` blocks in the HTML output. Spec §34 marks `mermaid-cli` as optional. Two viable paths: (a) build-time render via `mermaid-cli` (Node 22 + headless Chromium underneath; same install pain as Item 2), or (b) lazy-load `mermaid.js` browser-side and inject a `<script>` — but that breaks the "0 external refs" guarantee unless we bundle mermaid.js inline (~600 KB).
**Value rationale (2)**: Diagrams are nice for teaching but the user's current corpus (Sanipro courses, instructor-pack distribution) is text-heavy. The instructor will live without diagrams in v1.1.
**Cost rationale (4)**: Either path is heavy. Build-time means the same Chromium dep as Item 2. Browser-side means inlining ~600 KB of mermaid.js into every HTML export (multiplying export bundle size by ~2). Neither is a small dep.
**Risk**: Medium. Build-time path inherits Item 2's Chromium risk; browser-side path inflates exported HTML and complicates the "self-contained, no internet" gate.
**Pros**:
- Closes a spec-listed optional feature.
- Visually striking when used.
**Cons**:
- Heavy dep for a feature the current corpus doesn't use.
- Conflicts with the "students can read the source" principle (mermaid.js minified is opaque).
**Recommendation**: **DEFER TO v1.2+ or community contribution**. Re-evaluate when there's a real teaching script that needs a diagram.
**If shipped**: Slice with two paths to choose from; default to lazy-load browser-side with mermaid.js bundled. ~3 days, ~400 lines plus 1 large dep.

### Item 4 — Live OTLP/HTTP collector — `logbook otel start/stop/status`

**Source**: iter4 deferred from scope (iter4 used file-based OTLP-JSON only); iter5 roadmap.
**Spec reference**: §15 (OTel nativo).
**Description**: §15 describes a local OTel collector listening on `127.0.0.1:4318/v1/traces` for live ingest. Iter4 shipped only the file-based normalizer (`logbook ingest otel <file>`). The live collector would be a long-running daemon.
**Value rationale (1)**: For an instructor's workflow this is invisible. The use case is "developer is running an agent with OTel exporters and wants real-time ingest". The instructor reviews completed sessions, not live ones. The file-based path already covers post-hoc analysis.
**Cost rationale (4)**: A long-running HTTP server inside what is currently a one-shot CLI is a structural shift. State management, signal handling, port collision, pid file, status command, the test surface for a daemon is significant. ~600 lines, ~5 days, plus careful security review (the spec §31 rule "no outbound network" extends to "no inbound network without explicit user opt-in").
**Risk**: Medium. New attack surface (a listening socket). Tests need a port-reservation strategy.
**Pros**:
- Closes a spec §15 surface.
- Useful for the "I want to debug an agent live" persona.
**Cons**:
- Wrong persona for the instructor use case.
- Significant maintenance (daemon lifecycle).
- Crosses the "no listening socket by default" invariant.
**Recommendation**: **DEFER TO v1.2+**. Revisit when a developer-persona user shows up.
**If shipped**: Three slices (server + lifecycle + tests). ~1 week.

### Item 5 — SEA distribution binary

**Source**: iter5 roadmap.
**Spec reference**: §35 ("Distribución: pnpm como gestor primario; SEA opcional"), §36 iter5 row ("Distribución SEA opcional").
**Description**: Node 22's Single Executable Application support lets us produce a `logbook` binary that ships without requiring Node installed. Marked **opcional, no DoD gate** in the spec.
**Value rationale (2)**: For an instructor who already has pnpm and Node installed (the documented install path), zero value. For a future user who wants `curl | sh`-style distribution, real value — but that user does not exist yet.
**Cost rationale (4)**: Node SEA is still maturing in Node 22. It works but has rough edges around `better-sqlite3` native bindings, `@modelcontextprotocol/sdk` stdio transport, and Ink's terminal initialization. Realistic effort to get a clean macOS + Linux + Windows binary working: ~5 days, plus CI infra to build and publish.
**Risk**: High. The bundled CJS today already depends on `better-sqlite3` failing gracefully in tmp dirs (iter2 T10b.D2). SEA wraps that into a snapshot, which changes the failure mode.
**Pros**:
- Removes the Node prerequisite from install instructions.
**Cons**:
- Wrong target audience for v1.1.
- High maintenance — Node SEA semantics drift between minor releases.
- `better-sqlite3` native binding is the elephant in the room (see Item 18).
**Recommendation**: **DEFER TO v1.2+ permanently or until non-Node-dev users appear**.
**If shipped**: Multi-slice; one platform at a time. ~1 week per platform.

### Item 6 — `sessionStart` caching via `state.lastSummary`

**Source**: iter4 deferred; iter6 archive J.5.
**Spec reference**: §20 (memoria persistente entre sesiones).
**Description**: The `SessionStart` hook today rebuilds the ≤ 120-token summary on every Claude Code session start by reading recent events. Caching the rendered summary in `state.lastSummary` and only regenerating when events change would shave the hook latency further.
**Value rationale (1)**: The hook already runs at p95 = 49.5 ms (iter5 verify) — 4× under the 200 ms budget. No user perceives the difference between 49 ms and 30 ms. This is a micro-optimization the instructor never sees.
**Cost rationale (2)**: Add a hash of the last N events, compare to cached hash on read, regenerate if different. ~150 lines, ~1 day.
**Risk**: Low. Cache invalidation logic is the only landmine; if it gets stale, the summary lags behind reality.
**Pros**:
- Marginal perf improvement.
**Cons**:
- Adds complexity to a working hot path for no user-visible gain.
- Risk of stale-summary bugs.
**Recommendation**: **DEFER TO v1.2+ or skip**. The hook is already fast enough; cache invalidation has historical failure modes worse than the perf gain.
**If shipped**: Single slice with strict invariants (hash includes last event timestamp).

### Item 7 — Animated spinner in DoingScreen

**Source**: iter6 S3.
**Spec reference**: not in spec (TUI polish).
**Description**: The DoingScreen (shown during install / build / export / doctor / uninstall) currently renders a static `●` character. An animated spinner (`⠋ ⠙ ⠹ ⠸ ⠼ ⠴ ⠦ ⠧ ⠇ ⠏` cycling on a 100 ms interval) would signal liveness during multi-second operations.
**Value rationale (3)**: The instructor demos LogBook in front of an audience. A static dot looks like the program froze; an animated spinner looks like the program is working. Small but real teaching-context value.
**Cost rationale (1)**: Add a `useEffect` + `setInterval` inside `DoingScreen`, cycle through a 10-frame array, clear interval on unmount. ~30 lines, 1 unit test asserting the frame index advances. Half a day at most.
**Risk**: Low. Ink already supports `useEffect`; the rest is a counter modulo frames.length. The only correctness concern is making sure the interval is cleared on unmount to avoid React warnings.
**Pros**:
- Visibly improves the live demo experience.
- Trivial to implement.
**Cons**:
- Adds a useEffect to a previously pure component (slightly harder to test).
**Recommendation**: **SHIP IN v1.1**.
**If shipped**: Single PR, one slice, ~30 lines + test.

### Item 8 — `retry: 2` in vitest config for `mcp-rate-limit.test.ts`

**Source**: iter3 W (S1 closure attempt with sleep bump), iter4 carry, iter5 W1, iter6 W1 (4× carry).
**Spec reference**: not in spec.
**Description**: The `mcp-rate-limit > 21st call should return -32000` test is timing-sensitive. It uses `Promise.all` to fire 21 calls inside a 1 s sliding window and asserts the 21st returns the rate-limit error. Under parallel load this occasionally flakes. The sleep was bumped to 1300 ms in iter3, but flakiness persists under heavy parallel runs. The simplest closure is to add a per-test retry in `vitest.config.ts` (or scope it to that file via `test.retry`).
**Value rationale (3)**: CI flakes silently erode trust in the test suite. The instructor will not run CI, but they will pull updates; a flake during their first `pnpm test` after a fresh clone is a bad first impression. Closing this is "quality signal hygiene".
**Cost rationale (1)**: Two-line change in `vitest.config.ts` (or use the local `retry` option in the test file). ~5 minutes.
**Risk**: Low. `retry: 2` masks the flake without papering over genuine bugs because the flake has been independently re-verified across iter3 / iter4 / iter5 / iter6 as a timing artifact, not a logic regression.
**Pros**:
- Closes a 4-iteration carry warning.
- Trivial cost.
- Reduces support-load fingerprint.
**Cons**:
- A retry mask is not a root-cause fix. The "right" fix would be to inject the window size into the rate limiter and use a 50 ms window in tests. That's Item-8-extended, scope ~30 lines.
**Recommendation**: **SHIP IN v1.1**. While there, do the deeper fix: inject window size and use 50 ms in tests instead of `retry: 2`. Five-minute closure, real fix.
**If shipped**: One slice. Inject `windowMs` constructor arg into the rate limiter; default 1000, tests use 50. Drops the `await sleep(1300)` calls. Net result: rate-limit tests run in ~200 ms total, no retry needed.

### Item 9 — Extend `--safe` to `logbook build`

**Source**: iter5 suggestion.
**Spec reference**: §10 (build), §30 (seguridad y privacidad).
**Description**: Today the `--safe` flag works on `logbook export html` and `logbook export instructor-pack`. `logbook build` (which writes `logbook/index.md`, `timeline.md`, `errors-and-lessons.md`, etc.) does not accept `--safe`. If an instructor wants the source-of-truth artifacts inside the repo to be sanitized (paths, usernames, emails) without going all the way to HTML, they have no path.
**Value rationale (4)**: For an instructor who commits `logbook/` to the course repo (which is the canonical distribution mechanism in the spec), having the build pipeline emit redacted artifacts directly is a significant workflow improvement. Without it, the instructor exports HTML, then has to manually reconcile the redacted HTML against the unredacted Markdown.
**Cost rationale (2)**: `redact/anonymize.ts` and `export/safe.ts` already exist (iter4). The work is wiring `--safe` into `src/cli/commands/build.ts`, threading it through `src/generate/*.ts`, and applying the same redaction transform before writing each generated file. ~200 lines, 1–2 days.
**Risk**: Low. The redaction transform is well-tested (27 unit + 10 integration tests from iter4). The only risk is that `--safe` becomes the default in user habits and they lose the unredacted version — which we mitigate by making `--safe` explicitly opt-in.
**Pros**:
- Closes a real workflow gap.
- Reuses fully tested code.
- Aligns build output with export output.
**Cons**:
- One more flag in `logbook build` (already has --json and others; surface stays manageable).
**Recommendation**: **SHIP IN v1.1**.
**If shipped**: One slice in `src/cli/commands/build.ts` + threading the flag into each generator. Tests in `tests/integration/cli-build-safe.test.ts`.

### Item 11 — Real tokenizer

**Source**: iter1 deferred; mentioned across iter2–iter5 as "doctor --measure uses chars/4 heuristic".
**Spec reference**: §23.4.
**Description**: `doctor --measure` currently estimates tokens via `chars / 4`. The spec calls for "real tokenizer". Candidates: `@anthropic-ai/tokenizer` (correct for Claude), `tiktoken` (correct for OpenAI), or both routed by provider.
**Value rationale (2)**: The 499-token reading is conservative; the actual Claude token count is likely a few tokens lower. The user gains slightly more accuracy but the HARD GATE (≤ 500) is already passing comfortably. The risk is that the real tokenizer reports *higher* counts and the gate trips. Low probability but non-zero.
**Cost rationale (2)**: One dep, one wrapper in `src/core/token-measure.ts`, update tests. ~150 lines, 1 day.
**Risk**: Low logic risk, but real *political* risk: the chars/4 heuristic has held the 500-token gate for 6 iterations. If the real tokenizer reports 510, we either lower the gate's content or raise the gate — both require a spec amendment.
**Pros**:
- Accurate measurement.
- Spec compliance.
**Cons**:
- Could surface a budget breach in installed content.
- Pulls in a Claude-pinned dep (`@anthropic-ai/tokenizer` ties to Claude versions).
**Recommendation**: **DEFER TO v1.2**. Pair with a budget re-baseline if the new numbers shift.
**If shipped**: Single slice with dual run (chars/4 AND real tokenizer) reporting both for one release before switching the gate. Eases the transition.

### Item 13 — Hash-shape filter expansion (base64-encoded hashes)

**Source**: iter1 S2.D5 follow-up.
**Spec reference**: §30 (redacción).
**Description**: The hash-shape filter currently exempts 32/40/64/128-hex character strings from secret-redaction false-positives (SHA-1, SHA-256, SHA-512). Base64-encoded hashes (e.g., 44 characters with `+/=` padding) are not exempted, so a base64 SHA-256 of a string could be redacted as a secret.
**Value rationale (2)**: Real concern when the user's logs contain base64 hashes (Git, Subresource Integrity, etc.). The instructor's corpus today is plain Markdown; the risk is low. Becomes higher value if events start including base64-encoded artifact references.
**Cost rationale (2)**: Add a regex for base64 hash shape (44 chars, ends in `=` or two `==`), add negative-case tests. ~100 lines, 1 day.
**Risk**: Low. Pure additive filter.
**Pros**:

## Section 6 — Risk and trade-offs

LogBook v1.0 + v1.1 (this proposed cut) stays small enough that students can read the source. That is a feature, not an accident: spec §3 ("filosofía y posicionamiento") and the CLAUDE.md root constraint both treat readability as a design constraint. Every dep we add competes against that. `rehype-slug` (Item 1) is pure-functional and pulls in one transitive dep (`github-slugger`); that's an acceptable cost. Puppeteer (Item 2) is not.

The maintenance trade-off is sharper. Six iterations of the MVP pinned a specific stack: Vercel AI SDK, Claude Agent SDK, MCP SDK, citty, valibot, Ink, unified/remark/rehype, better-sqlite3. Each of those will drift in 2026; every new dep we add today is another upgrade obligation in 6 months. The recommended v1.1 cut adds exactly one new dep (`rehype-slug` plus its transitive `github-slugger`). Items 2, 3, 5, 17, 18 each add heavy native or platform-coupled deps; v1.1 leaves them all out.

The "ship what's needed for actual use" principle dominates the recommendation. The user is teaching with this in a few days. Items that make the instructor's first week easier (1, 7, 8, 9, 20) are in. Items that round out the matrix without affecting that week (4, 6, 10, 12, 17, 18, 22, 23) are out. Items that look attractive but cost a week or more (2, 3, 5, 15) wait until the production data tells us they matter.

If forced to pick a single item: Item 1. The TOC in `instructor-pack.html` is the canonical artifact the instructor hands out. Today it looks broken on first scroll. That's the v1.1 priority.
